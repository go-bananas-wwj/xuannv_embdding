# 玄女科技 BP 后续章节：架构原理与范式革新页面计划 v0.1

## 章节定位

前 9 页负责让投资人理解“玄女是谁、为什么是遥感行业的 GPT 时刻、为什么需要中国自己的遥感数据通用嵌入底座”。

能力案例页负责用哈尔滨新区、海淀区、雅江区域证明“玄女已经能在真实场景中交付能力”。

## 页面 1：从案例回到本质：玄女改变的是遥感生产方式

**目标**

把前面的哈尔滨、海淀、雅江案例收束成一个判断：玄女不是多做了几个遥感应用，而是在改变遥感应用的生产方式。

**标题建议**

从案例到范式：遥感应用正在从项目制走向底座化

**核心表达**

传统遥感交付是“一个需求、一套数据、一轮标注、一个模型、一份报告”。玄女的路径是“先生成统一地理嵌入，再把同一套表征复用到变化检测、分类、检索、问答和报告生成”。

**页面内容**

- 左侧：传统遥感项目制链路
  - 数据采购
  - 数据预处理
  - 专家标注
  - 单任务建模
  - 制图报告
  - 下一个任务重新开始
- 右侧：玄女底座化链路
  - 多源观测输入
  - 统一地理嵌入
  - 多任务头 / 智能体 / 报告工作流
  - 新任务低成本复用
- 底部一句结论：
  - 玄女的核心不是“做更多遥感项目”，而是把遥感项目沉淀为可复用的地理智能基础设施。

## 页面 2：技术范式革新：从图像学到自学习

**目标**

讲清楚“传统遥感为什么慢”，以及玄女为什么不只是提高效率，而是换了技术范式。

**标题建议**

技术范式：从“图像学 + 专家规则”到“多模态自学习”

**核心表达**

传统遥感依赖人类显性知识：物理公式、专家经验、人工规则和单任务标注。玄女依赖模型从多模态、多时序观测中自学习隐性规律，把知识沉淀进统一地理嵌入。

**页面内容**

做成左右对照，不要写大段文字。

左侧：传统图像学范式

- 专家先定义特征
- 人工融合多源数据
- 单任务训练
- 换任务需要重做
- 交付周期长、边际成本高

右侧：玄女自学习范式

- 多模态数据自动对齐
- 模型学习时空连续规律
- 地理嵌入沉淀隐性知识
- 下游任务轻量适配
- 一次表征，多次复用

**建议可视化**

左侧用“人工规则/专家/单任务模型”的流程图，右侧用“多模态数据 → 自监督学习 → 地理嵌入 → 多任务复用”的流程图。

**需要避免**

不要写“边际成本趋近于零”这种过满表述，可以改为“边际交付成本显著下降”。

## 页面 3：核心原理：把地球观测转成可计算的地理嵌入

**目标**

解释玄女模型最核心的技术思想：不是直接存图、看图、判图，而是把地球观测转化为可检索、可复用、可计算的嵌入。

**标题建议**

核心原理：从“数字地球”到“嵌入地球”

**核心表达**

数字地球主要解决“把数据存下来”；嵌入地球解决“把地理知识编码出来”。玄女将同一地理位置的多源、多时序观测压缩为统一表征，让下游任务可以直接调用。

**页面内容**

三段式：

1. 数字地球
   - 存储遥感影像、矢量、DEM、专题数据
   - 数据仍然分散、稀疏、难以直接使用
2. 地理嵌入
   - 对同一地理位置形成统一表征
   - 融合时间、空间、频谱、语义信息
3. 下游调用
   - 变化检测
   - 地物分类
   - 检索
   - 预测
   - 报告生成

**建议可视化**

左：很多遥感图层堆叠。

中：压缩成统一的地理嵌入立方体或向量格网。

右：多任务出口。

**可用表达**

GPT 先把文本转成 token，才能理解、检索和生成；玄女先把地球观测转成地理嵌入，才能做变化检测、分类、预测和问答。

## 页面 4：模型架构：多模态输入，统一嵌入，多任务解码

**目标**

给投资人一个足够清楚但不过度学术的模型架构图，说明“为什么它可以支持多任务”。

**标题建议**

模型架构：多模态观测进入统一地理嵌入空间

**核心表达**

玄女底座将光学、SAR、多光谱、高光谱、DEM、气象、反演指数等多源数据输入统一时空编码器，生成地理嵌入；不同下游任务通过任务头或解码器调用同一套表征。

**页面内容**

从左到右：

1. Input：多模态观测
   - 可见光
   - SAR / 微波
   - 多光谱 / 高光谱
   - DEM
   - 气象
   - 指数与反演产品
2. Encoder：时空多模态编码器
   - 时空对齐
   - 跨模态融合
   - 时序学习
3. Embedding：统一地理嵌入
   - 像素级 / 格网级表征
   - 可复用、可检索、可比较
4. Decoder / Task Head：下游任务
   - LUCC
   - DEM/高程回归
   - 变化检测
   - 地物分类
   - 检索问答
   - 报告生成

**建议可视化**

用清晰的架构图，不要复杂论文图。输入图标在左，编码器在中，嵌入立方体在中右，多任务出口在右。

**参考来源**

## 页面 5：训练路线：从自监督学习到场景增强

**目标**

解释玄女不是靠每个项目重新训练，而是先做通用预训练，再用少量样本和场景知识快速适配。

**标题建议**

总体技术路线：通用预训练 + 少样本场景增强

**核心表达**

玄女的训练路线是“先学习地球通用表征，再面向具体业务快速适配”。通用底座负责学习地理规律，场景增强负责把具体业务定义转成可执行任务。

**页面内容**

一条从左到右的路线：

1. 多源数据汇聚
   - 国产卫星
   - 公开遥感
   - 地面/辅助数据
2. 自监督预训练
   - 时序连续性
   - 跨模态补全
   - 空间一致性
   - 表征对齐
3. 通用地理嵌入
   - 统一底座
   - 长期复用
4. 场景增强
   - 少量标注
   - 业务规则
   - RAG / 智能体
5. 交付
   - 任务结果
   - 指标统计
   - 报告与接口

**建议可视化**

用“主干 + 分支”的路线图：主干是通用底座，分支是哈尔滨/海淀/雅江等场景适配。

**需要谨慎**

如果还没有完整验证的训练目标，不要写得过于绝对。可以写“训练目标包括/可包括”。

## 页面 6：产品架构：通用底座与场景模型并行

**目标**

解释玄女不是只卖一个模型文件，而是一个可被政府、企业、开发者调用的产品化底座。

**标题建议**

产品架构：一个通用底座，支撑多类场景应用

**核心表达**

通用底座负责沉淀地理表征能力；场景模型负责适配行业任务；上层智能体和报告工作流负责把模型能力转成用户可直接使用的产品。

**页面内容**

三层架构：

1. 数据层
   - 多源遥感
   - DEM / 气象 / 指数
   - 本土标签与反馈数据
2. 底座层
   - 多模态时空编码器
   - 地理嵌入库
   - 向量检索
   - 任务头 / 解码器
3. 应用层
   - 政务监测
   - 城市治理
   - 水电站监测
   - 智能体报告
   - API / SDK

**建议可视化**

## 页面 7：业务范式革新：从项目交付到基础设施调用

**目标**

把技术路线翻译成商业语言：为什么这个技术会改变遥感公司的盈利方式和交付方式。

**标题建议**

业务范式：从专家项目制，到底座调用式交付

**核心表达**

传统遥感公司靠专家、数据采购和项目交付收费，交付周期长、边际成本高。玄女把数据处理、模型能力、任务适配和报告生成沉淀到底座中，使交付从“项目制”转向“调用式”。

**页面内容**

左右对比：

传统项目制：

- 一个项目一套流程
- 专家和标注驱动
- 交付周期长
- 收入增长依赖人力扩张
- 新任务重复投入

底座调用式：

- 一套嵌入多次复用
- 模型和数据闭环驱动
- 交付周期缩短
- API / License / 智能体订阅
- 新任务轻量适配

**底部结论**

玄女的商业价值在于把遥感能力从“高成本定制服务”变成“可持续调用的基础设施能力”。

**参考来源**

上传 PPT 第 9 页“业务范式革新”、第 14 页“三层架构”、第 24 页“一句话总结”。
