## 诊断结论

1. 当前 OSM 下游头存在明显概率校准问题：4 个任务的验证阈值都在 0.95 到 0.97 之间，说明 `0.5` 不是可用阈值。
2. `F1@0.5` 明显低于 `F1@val_thr`，说明模型常把大量区域打成高概率，阈值稍低时前景会泛滥。
3. 对稀疏目标任务，后续汇报不能把 ROC-AUC 作为主指标。主指标应改为：`F1@0.5`、`F1@val_thr`、`mIoU`、预测正类比例、TP/FP/FN 可视化。
4. 后续训练/测评流程应固定生成诊断图，避免只用 AUC 或只展示好看的样例。

## 后续建议

- 下游头训练加入更强的类别不平衡处理，例如 Dice/Focal、Tversky、正负样本采样控制。
- 增加概率校准评估，例如 ECE、reliability diagram、固定正类比例阈值。
- 对道路/建筑任务分别处理：道路应强化细线结构连续性，建筑应强化块状边界和面积一致性。
- 汇报时同时放 AUC 与阈值后结果，明确说明 AUC 只是辅助排序指标。

## 2026-06-27 追加：红色预测图与效果偏弱原因分析

用户要求预测前景也使用红色、背景使用白色，便于和 GT 直接对比。脚本已更新，当前 `osm_diagnostic_sheets` 目录下的诊断图已经覆盖生成：

- `Prediction Prob`：白到红的概率热图，越红表示概率越高。
- `Pred >= 0.50`：白底红色二值预测。
- `Pred >= val_threshold`：白底红色二值预测。
- `OSM GT`：白底红色真实标签。

### 为什么效果不好

从当前 quick OSM 任务看，问题不是单一原因，而是几个因素叠加：

1. 下游头明显偏向“多预测前景”。配置里 `pos_weight=150.0`，同时使用 `bce_dice_tversky` 和 `tversky_beta=0.7`。这套损失会强烈惩罚漏检，模型为了提高召回，容易把大量背景也预测成前景。
2. 概率校准很差。4 个任务的验证阈值都在 0.95 到 0.97，说明 `0.5` 阈值基本不可用。样例统计中，`haidian_road_osm` 的 `Pred >= 0.5` 平均正类比例达到 1.0，几乎整图都被预测成道路前景。
3. OSM 标签天然有噪声。OSM 道路/建筑来自矢量数据，和遥感影像存在位置偏差、时间差、缺失和过时问题。道路尤其容易因为线宽缓冲、道路等级、遮挡、树冠覆盖导致 mask 和影像不完全对齐。
4. 当前 P1B embedding 主要来自变化/稀疏监督训练，不是专门针对建筑边界、道路细线结构、水体边缘做的密集语义监督。因此 embedding 能提供一定排序信息，AUC 会有提升，但下游头很难直接得到干净边界。
5. 低分辨率 128x128 输出会损失细线目标。道路这类窄目标在 10m 等效分辨率下很容易变成粗糙块状 mask，导致 FP/FN 都高。
6. 当前 quick 测评只训练 30 epoch，更多是快速验收，不是针对 OSM 下游任务充分调参后的结果。

### 后续修复方向

- 降低或自适应设置 `pos_weight`，不要固定 150；按任务真实正负比例和验证集表现调节。
- 单独为建筑/道路任务设计 loss：建筑可用 BCE + Dice/Tversky；道路增加 skeleton/centerline 或 clDice 类结构约束。
- 把报告主指标改成 `F1@0.5`、`F1@val_threshold`、`mIoU`、预测正类比例、FP/FN 可视化，AUC 只作为辅助排序指标。
- 对 OSM 标签做质量控制：去掉很小碎片、按影像分辨率调整道路 buffer、过滤低可信类别、抽样检查矢量与影像配准。
- 对概率输出做校准：温度缩放、验证集阈值、固定预测面积比例阈值，并输出 reliability/ECE。
